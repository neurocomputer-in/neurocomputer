# NeuroFramework

## 1. Core Claim

NeuroFramework is not just a codebase structure.  
It is a **modular cognition framework** that is already behaving like a proto-language.

The core idea is simple:

- everything important in intelligent computation is represented as a **neuro**
- neuros are composable
- composition builds higher-order behavior
- higher-order behavior forms agents, agencies, and full software systems

This is why NeuroFramework is positioned as a path toward AGI-grade engineering:
not one giant model, but a programmable ecosystem of interoperable cognitive units.

## 2. Neuro as the Universal Unit

A neuro is a typed, composable runtime unit with:

- identity (`conf.json`)
- behavior (`code.py`)
- optional language surface (`prompt.txt`)

Neuros are not only "skills". In this framework, **model, prompt, instruction, context, memory, workflow, and agent** are all neuro-level abstractions.

That means the system uses one unifying ontology for:

- thinking
- planning
- tool use
- memory
- model invocation
- behavior policies

This unification is the foundation of neurogramming.

## 3. All Framework Components (Conceptual)

NeuroFramework has these first-class component classes:

1. **Model neuros**  
   Interface to LLMs and model backends. Model choice becomes composable behavior, not hardcoded infrastructure.

2. **Prompt neuros**  
   Prompt blocks and prompt composers as reusable modules.

3. **Instruction neuros**  
   Rules, policy, and tone as explicit behavioral constraints.

4. **Context neuros**  
   Context slicing and assembly as programmable operations.

5. **Memory neuros**  
   Store, recall, extraction, categorization, consolidation, and layered memory logic.

6. **Skill neuros**  
   Concrete capabilities (system operations, integrations, utilities, domain tools).

7. **Workflow neuros**  
   Sequencing, DAG orchestration, parallel fan-out, and iterative tool loops.

8. **Agent neuros**  
   Role-specialized orchestration units built from other neuros.

9. **Code neuros**  
   Read/write/diff/patch/plan/scan operations for self-modification and coding workflows.

10. **Library/namespace components**  
    Registry, taxonomy, and discoverability so neuros can be reused like language/library primitives.

## 4. Why This Is Language-Like

NeuroFramework is evolving toward a language because it already has language properties:

- **vocabulary**: neuro kinds and namespaces
- **grammar**: how neuros compose in flows (sequential, DAG, parallel, looped)
- **semantics**: typed behavior contracts and execution rules
- **runtime**: factory + brain + flow executor + state + event bus
- **standard library**: built-in neuros and reusable domain neuro libraries

In other words, "programming" here becomes:

- selecting neuros,
- composing neuros,
- constraining neuros,
- and evolving neuro libraries.

## 5. Computational Expressiveness (Turing-Complete Direction)

In principle, the framework is computationally universal because it supports:

- state
- branching/conditionals
- iterative execution (`tool_loop`, flow orchestration)
- unbounded composition
- external I/O and tool interaction

That is the practical basis for claiming **Turing-complete direction**:
the framework can express general computation through neuro composition, not only prompt chaining.

## 6. Neuro Libraries and Namespaces

Neuros are organized as libraries with kind namespaces (for example):

- `skill`
- `prompt`
- `memory`
- `context`
- `model`
- `instruction`
- `agent`
- `library`

This matters because it enables:

- discoverability
- reuse
- compositional authoring
- team-level interoperability
- long-term ecosystem growth

The end-state is a community and internal ecosystem where anyone can create a new neuro by combining existing neuro libraries and extending namespaces.

## 7. Neurogramming IDE (3D LEGO for Coding)

The Neurogramming IDE is not an accessory; it is the interaction model for this framework.

Desired behavior:

- visualize neuros as graph-native programmable units
- inspect contracts, code, and prompt surfaces together
- compose workflows visually and textually
- validate/save/rollback safely
- reuse blocks across projects/agencies

This is "LEGO for coding":
not editing one giant file, but building systems by assembling intelligent components with explicit contracts.

## 8. AGI Trajectory

NeuroFramework's AGI relevance is architectural:

- modular cognition instead of monolith
- explicit memory + context + instruction + model boundaries
- programmable orchestration over opaque end-to-end prompting
- self-improving libraries through reusable neuro abstractions

The framework is designed so intelligence can scale by composition:
from one neuro, to workflows, to agents, to agencies, to ecosystem-level cognition.

## 10. Design Principle

If a capability matters for intelligent behavior, it should be representable as a neuro.  
If neuros are composable and reusable, the system becomes programmable intelligence.  
If programmable intelligence is scalable, we get a credible path toward AGI systems engineering.
